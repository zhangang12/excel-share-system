## 二、开发工作流约定(Claude Code 会话)

### 分支与推送

- 开发分支:当前用 `claude/elegant-newton-vibso6`,同时推 `main`(用户习惯直接从 main 部署)。
  新会话会分配新的 claude/* 分支,沿用"开发分支 + main 双推"即可。
- **推送方式**:沙箱默认的 git 代理是只读的(push 返回 403)。可用方案是 PAT 直推:
  1. 仓库根目录建 `.env.local`(已在 .gitignore),内容一行:`GITHUB_PAT=github_pat_xxx`
     (**接手者用自己 GitHub 账号新生成 PAT**,旧 PAT 不随仓库/容器转移,也不应共用);
  2. 推送命令模式:
     ```bash
     set -a; source .env.local; set +a
     GIT_CONFIG_COUNT=0 GIT_TERMINAL_PROMPT=0 \
       git push "https://x-access-token:${GITHUB_PAT}@github.com/<owner>/excel-share-system.git" \
       "HEAD:main" "HEAD:refs/heads/<开发分支>" 2>&1 | sed -E 's#x-access-token:[^@]*@#x-access-token:***@#g'
     ```
     (末尾的 sed 是为了不在日志里泄露 token)
  3. 推送前检查快进安全:`git merge-base --is-ancestor origin/main HEAD`。
- **绝对不要**把 `.env.local`、PAT、任何密钥提交进仓库或写进代码/文档。

### 修改后的验证习惯(教训换来的,别省)

- 前端:`cd frontend && npx vue-tsc --noEmit` 必须过。
- 后端:**新写/改动的接口必须实际调用一遍**,不能只靠类型检查。
  (教训:`/sales/equipment-names` 曾引用不存在的 `SalesLedger.name` 字段上线才炸——
  设备名在 `Project.name` 上。)
- **涉及 SQL/schema 的改动要在真 Postgres 上验证**,不能只跑 SQLite(见第三节第 1 条)。
  沙箱里起真 PG:`service postgresql start`,已有测试用户 `pms`/`pms`,
  可 `su - postgres -c "psql -c 'CREATE DATABASE xxx OWNER pms;'"` 建临时库。
- 后端测试在 `backend/tests/`,可参考 `test_e2e_full_lifecycle.py` 的写法。

### 部署提醒(每次推完代码要告诉用户)

- 用户自己在服务器上执行,标准动作:

## 三、生产环境踩坑记录(按重要性排序)

1. **SQLite vs Postgres 差异是本项目最大的隐性雷区**(沙箱开发用 SQLite,生产是 Postgres):
   - SQLite 不校验 `VARCHAR(n)` 长度,PG 严格校验。踩过:`material_dict.dtype VARCHAR(16)`
     存 `'supplier_category'`(17 字符)在生产炸掉。已修(现为 32)+ 生产已手工
     `ALTER TABLE material_dict ALTER COLUMN dtype TYPE VARCHAR(32);`(一次性,勿重复)。
   - PG 严格要求 ORDER BY 列出现在 GROUP BY 或聚合中(GroupingError),SQLite 不管。
     踩过:`oa_summary` 按 `Department.sort_order` 排序但没进 GROUP BY,生产 500。
   - **规则:任何带 GROUP BY/ORDER BY 组合、或新增字符串列的改动,上真 PG 跑一遍。**
2. **时区**:容器日志是 UTC,服务器/诊断包时间是 CST(UTC+8)。对日志时间时记得 +8
   (日志里 15:15 = 北京时间 23:15)。
3. **healthcheck 噪音**:`pg_isready` 不带 `-d` 时 dbname 默认=username,曾造成日志刷
   378 条 `FATAL: database "pms_prod" does not exist`。docker-compose 两个文件的
   healthcheck 都已加 `-d ${POSTGRES_DB}`,改 compose 时别丢掉。
4. **诊断脚本必须带超时**:`ops/diagnose.sh`/`health-check.sh` 里所有 docker/psql 调用
   都包了 `timeout -k 3 <N>`、curl 带 `--connect-timeout/--max-time`。
   原因:被诊断的服务卡死时,不带超时的诊断脚本会跟着一起卡死。新加诊断命令保持这个模式。
5. **中文 PDF**:服务端 PDF 用 reportlab 内置 CID 字体 `STSong-Light`,无需外部字体文件。
   沙箱里 pdftoppm 渲染中文报 "Missing language pack for Adobe-GB1" 是渲染器缺
   `poppler-data`,不是 PDF 本身的问题。
6. **金额字段全是 float**:增量清单 M10 就标注过应改 Numeric/分存储,一直未做。
   做财务精算类功能时注意累计误差,新表建议直接用 Numeric。

## 四、生产排障流程(用户报"服务器报错"时)

1. 让用户在服务器跑 `bash ops/diagnose.sh`,会产出 `pmsdiag<时间戳>.tar` 诊断包
   (含各容器日志、健康检查、可能附带报错截图 base64)。
2. 拿到包后:解包 → 后端日志里 **grep `未捕获异常`**(这是 500 traceback 的可靠锚点),
   截图类文件是 base64,解码出来看。
3. 定位后:本地复现(优先真 PG)→ 修 → 推 main → 提醒用户 pull + rebuild。
4. 常用:`ops/logs.sh`(看日志)、`ops/db-shell.sh`(进库)、`ops/health-check.sh`(体检)。
