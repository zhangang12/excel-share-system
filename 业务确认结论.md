# 业务确认结论（开发依据 · 定稿）

> 来源：业务方填写的《业务口径确认结果》+《设计部电工部生产部功能需求文档》+ 当面补充。
> 作用：作为后续开发的**唯一口径基准**；与《需求分析》《技术方案》《仓库管理功能设计》配套。
> 红线（贯穿全部）：**不改现有生产页面与功能，只新增；确需调整老页面的，一律用"可逆开关"而非破坏式改动。**

---

## 〇、最关键的两条处置（先看）

1. **项目目录"制图开始/制图结束/制图用时"三列 → 列隐藏，不物理删除。**
   - 业务想法仍在变，故**不删列、不改表结构、不动历史数据**；新增一个"列显示配置"（管理层可配），把这三列设为隐藏，随时可恢复显示。
   - 制图的权威台账（开始/预计完成/完成/用时/完成效率）改由**设计部**维护。
   - 原"设计部制图开始**回传**项目目录"的逻辑**取消**（B3：不再回传）。
   - 红线符合性：仅"加一个隐藏开关 + 设计部新模块"，可逆、零破坏。

2. **上线方式：全量切换；存量项目（数十个进行中）也纳入新流程。**（H1/H2）

## 一、角色与权限（A）

| 项 | 结论 |
|---|---|
| A1 | 生产文员 / 仓库员 / 人事行政：**保留原职责，不参与新部门下单/接单** |
| A2 | 采购员合并（标准件+外协→采购部）：**存量用户自动并入采购部，并迁移其字段权限** |
| A3 | 各部门页面（设计/电工/生产…）可见范围：**仅本部门成员 + 管理层** |
| A4 | 生产部下单：**仅生产部主管可下生产单**（管理层不越权下生产单） |

> 新增角色（沿用技术方案）：电工、装配组1/2/3、采购部、物流发货部、设计/电工/生产部主管、仓库主管；并标记"消息推送人"。

---

## 二、下单与执行（B）

| 项 | 结论 |
|---|---|
| B1 | 同项目同部门**允许重复下单（新建一条任务）**。补充：**设计部可将原单号作废 → 管理层收到信息后可重复下单**；任务的项目编号**沿用项目目录编号**。 |
| B2 | 可下单范围：**仅"进行中"项目**（已完成/归档不可下单） |
| B3 | **不再回传**制图开始；制图开始/结束/用时**归入设计部**；项目目录三列**隐藏不删**（见〇.1） |
| B4 | 一个任务**单负责人**（装配组也是单负责人，不多人指派） |
| B5 | 进行中可改"开始/预计完成"时间，但**本人不能改、仅管理层可改** |

> 下单链路（据需求文档）：**销售下单 → 设计部/电工部负责人接单 → 在本部门页面分派给具体设计师/电工**；生产部由**生产部主管**直接下单选装配组。（详见"新增范围"销售部）

## 三、时间与效率口径（C）

| 项 | 结论 |
|---|---|
| C1 | 逾期/完成效率按**自然日**（沿用现状） |
| C2 | 预计完成**当天**完成 → **算按时**（done == due 不算逾期） |
| C3 | 预计用时为 0（开始=预计完成同日）→ **按 1 天计算** |
| C4 | 月度报表按**任务下单时间**计入当月 |
| C5 | 个人"按时率" = 按时数 **÷ 已完成任务数** |
| — | 完成效率 = 实际用时 ÷ 预计用时（≤100% 绿、>100% 红；数值越低越好） |

## 四、完成与下游（D）

| 项 | 结论 |
|---|---|
| D1 | 四张表"已上传"判定：**必须导入过 Excel**（仅手工加行不算） |
| D2 | 四表校验**只卡设计部完成**（电工/生产完成不校验四表） |
| D3 | 设计部完成"自选通知人"：**限相关角色、单选**；可选择**该角色下的具体人员** |
| D4 | 电工部：采购清单(Excel)**接单后**上传→采购；电路图(PDF)**完成时必传**→物流 |
| D5 | 物流发货**前置条件**：需**设计/电工/生产都完成**才可发 |

> 各部门产物：设计部完成传 说明书(Word)+铭牌(CAD)→物流；电工部如上；生产部无产物，仅通知物流。

## 五、物流与收货（E）

| 项 | 结论 |
|---|---|
| E1 | 一个项目**一次发货（一条发货待办）**，不分批 |
| E2 | 收货信息来源：**新建项目时录入**；由**销售/管理层新建项目时的销售角色**填写 |
| E3 | 生产部不产出图纸：**不需要**补"完工单"到物流三栏，**生产仅通知物流** |

## 六、企业微信与通知（F）

| 项 | 结论 |
|---|---|
| F1 | 系统用户 ↔ 企微成员：**管理页手动绑定 userid** |
| F2 | 通知频率：**即时单条**；逾期**每天提醒一次**（直到处理） |
| F3 | 站内消息形态：**铃铛 + 消息列表**即可（不强制已读/未读待办中心） |

## 九、本次新增范围（确认纳入，待设计）

> 以下来自《设计部电工部生产部功能需求文档》，超出原型与原《需求分析》，确认纳入后续设计：

1. **销售部（新一级菜单）**：每个销售人员有"下单"按钮；**销售人员只看自己的单**，销售主管 + 管理层看全部；新建项目时由销售填写内容（含收货信息 E2）。→ 这是下单链路的起点（销售→部门负责人分派）。
2. **钣金组（新一级菜单）**：设计部新增"上传 PDF 图纸文件包"并关联钣金组、推送钣金组人员；销售新建项目后钣金组收到项目信息、图纸包与项目关联；项目目录里的"钣金装配"表也挪到钣金组。
3. **仓库组（新一级菜单，两项功能）**：
   - ① **仓库管理**：物料出入库与实时库存 —— 已出《仓库管理功能设计》+ 原型 `demo/仓库管理-原型.html`。
   - ② **发货清单上传**：上传与项目目录编号挂钩的发货清单 → 推送物流发货部对应项目（复用附件+项目关联+通知物流机制）。
   - 设计部对仓库库存表**只读查看**。

## 十、实施红线与原则（必须遵守）

1. **只增不改**：新功能一律以"新菜单 / 新表 / 新接口 / 新组件"承载；不重构 `项目目录 / 项目详单 / 进度表 / 权限` 等现有页面。
2. **确需动老页面 → 用可逆开关**：如制图三列采用"列隐藏配置"，不物理删除、不改表结构、保留历史数据，随时可恢复。
3. **存量兼容**：迁移函数幂等、可重跑（沿用 `data_migration.run_all`）；不破坏既有数据。
4. **不阻塞**：企微凭证未到位时站内兜底；逾期定时任务多实例幂等。

## 十一、仍待确认（仓库管理专属，来自设计稿）

## 十二、流程评审追加决策（2026-06-08）

> 来源：《流程评审与问题清单》评审后业务拍板。

1. **钣金装配表**：保留为设计「四表校验」之一不变；**钣金组只读引用同一张表**（不物理移走，避免破坏四表校验）。
2. **部门先后依赖**：采用**软提示并行**——允许设计/电工/生产并行赶工，上游未完成仅在工作台给⚠提示，**不硬门禁**。
3. **项目「已完成/归档」**：系统**不自动**改项目状态，**由管理层在项目详情手动维护**（进行中/已完成/已归档）。
4. **销售下单**：销售**一次可勾选多个部门（设计+电工）**下单，各生成一条待派任务并分别推对应部门负责人；**生产**仍由生产部主管按节点单独下单。
5. **物流发货闸门**：设计/电工/生产**已下单的任务**全部完成才可发货（D5）；未下单的部门视为不涉及、不阻塞。
6. **完成可逆**：任务由已完成改回进行中时，**自动撤回**已推送给物流/采购/钣金的产物（保持下游一致）。
7. **部门页可见性**：各部门页仅本部门成员 + 管理层可见（A3）。
8. **生产→物流**：物流看板显示生产状态（无产物 E3），不要资料。
9. **仓库→物流**：物流看板新增「仓库发货清单」列承接。
10. **设计完成通知人**：限相关下游角色下的人员（D3）。
11. **逾期推送**：对应部门主管 + 抄送管理层。
12. **设计产物入口**：说明书 + 铭牌在「完成」弹窗上传 → 物流；**PDF 图纸包在「设计部工作台·接单后」上传 → 推钣金组**（上传在设计部、钣金组只负责下载）。
13. **作废重下**：重下 = 新任务，原单留痕作废、原资料不带入。
14. **设计看库存**：不进仓库菜单，设计页内嵌「查库存(只读)」。
15. **导出审批**：管理层导出免审，其余角色必审（G1/G2）。
16. **四表判定**：必须导入过 Excel 才算上传（手工加行不算 D1）。
17. **月度报表范围**：设计/电工/生产 + 销售下单量 + 仓库出入库量；**仅管理层可见**（其余角色含各部门主管均无权访问）。
18. **效率口径**：预计用时 0 按 1 天算（C3）；预计完成当天完成算按时（C2）。
19. **制图列处理**：项目目录与项目详情均**不展示**制图开始/结束/用时（**逻辑删除**：后端字段保留、UI 不显示、**无开关、业务无感知**）；制图改由设计部维护。
20. **企业微信绑定**：先做「用户↔企微 userid」手动绑定管理页（F1）。
21. **建项目入口**：编号全局唯一，销售下单 / 项目目录新建 两入口都可建。
22. **采购合并迁移**：脚本自动（幂等）把标准件/外协采购员并入采购部并迁字段权限（A2）。
23. **收货信息权威**：销售在下单/建项目时录入为准，物流可补充修正、留痕。

## 十三、销售下单补充：销售项目统计（来源《销售项目统计.xlsx》）

销售部由"下单列表"升级为"**销售项目统计台账**"。列：序号 / 项目编号 / 设备名称 / 客户单位 / 客户分类 / 销售 / 下单日期 / 交货日期 / 合同 / 金额 / 税票 / 发票情况 /（财务收款：预付 / 发货前付 / 发货款应收 / 尾款 / 发货日期 / 尾款日期）。

1. **项目编号自动生成**（按序号递增，如 2026-0XX，不手填）。
2. **客户分类**下拉：经销商 / 终端客户。
3. **合同**下拉：有 / 无（可改）。
4. **税票**下拉：13% / `/`（不开票）；**开票审批流**：销售员上传开票申请表 → **销售主管审批**（通过/驳回）→ 通过后推送**财务部（新增角色）** → 财务开票**上传发票，自动回传**销售订单（「发票情况」→ 0，发票可下载）。台账「发票情况」按状态显示：未开 / 待主管审批 / 待财务开票 / 已开票·0📎。
5. **表格可筛选**（按 客户分类 / 合同 / 关键词）。
6. 财务收款字段（预付 / 发货前付 / 发货款应收 / 尾款 / 发货日期 / 尾款日期）：已纳入**销售下单录入 + 台账列**；收款核销逻辑（按阶段算应收 / 到款登记 / 尾款提醒）待后续。
7. 销售台账中**项目编号仅展示、不链接**到项目目录（查看该项目流程用每行「🔍 全览」）。
8. **下单日期 = 合同签订日期**：销售下单表单**不填交货日期**；操作列「📄 上传合同」时填写**合同签订日期（= 下单日期）**与**交货日期** + 传合同附件，提交后「合同」标记为「有」、下单/交货日期自动回写台账。

> 原型已落：`demo/增量UI设计图.html` 销售部＝销售项目统计台账（自动编号 + 客户分类/合同/税票三下拉 + 开票申请 + 筛选 + 查看全览）。

## 十四、生产问题反馈流（装配组 → 设计师）

**装配组**在自己项目中提交**问题文字反馈** → **生产部主管审批**（通过/驳回）→ 通过后回馈给该项目的**设计师** → 设计师**接收（存档）**或**驳回** → 接收后**存档到该项目「问题反馈」列**（项目详情·部门协作 tab 可查）。

- 状态：待主管审批 → 待设计接收 →（设计师）已存档 / 设计驳回；（主管）主管驳回。
- 触点：装配组「生产部工作台·📝问题反馈」提交；生产主管「生产部·问题反馈审批」；设计师「设计部·📥待接收的问题反馈」接收/驳回；项目详情·部门协作「📝问题反馈存档」。
