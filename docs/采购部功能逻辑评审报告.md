# 采购部功能逻辑评审报告

> 评审日期：2026-07-21  
> 评审范围：采购管理全模块（供应商、采购明细、收货入库、请款付款、采购申请、前端权限）  
> 核心文件：`purchase_mgmt_router.py`（2800行）+ 关联模型/路由/前端

---

## 一、总体评价

采购部模块是系统最核心的业务模块之一，覆盖了「供应商管理 → 下单 → 收货 → 对账 → 请款 → 付款」完整链路。整体设计思路清晰，关键业务节点有内控校验，代码质量中上。

**亮点**：
- 删除保护链完整（对账状态→仓库流水→请款记录三道关卡）
- 自动对账 `_maybe_auto_reconcile` 条件严格不误触
- 内控职责分离（不能审批自己提交的请款单）
- 先收货后补价闭环
- 幂等设计优秀（防重复入库/冲红）

**主要风险**：
- 请款/付款金额无上限校验（可超额支付）
- 采购单号并发竞态
- 前端权限仅靠 username 硬编码

## 二、问题清单（按严重程度）

### 🔴 高严重度（2项）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| H1 | **付款金额无上限校验** | `purchase_mgmt_router.py:2276-2278` | `pay_payment_request` 直接接收 `paid_amount`，未校验是否超过请款金额或采购明细应付余额。财务可输入任意金额完成付款 |
| H2 | **请款金额无校验** | `purchase_mgmt_router.py:2150-2153` | `create_payment_request` 的 `requested_amount` 无校验，未检查是否超过关联采购明细的 `received_amount`（应付总额） |

**修复建议**：在付款接口增加双重校验：
```python
# 1. 付款总额不能超过请款金额
if paid_amount > pr.requested_amount:
    raise HTTPException(400, f"付款金额 ¥{paid_amount} 不能超过请款金额 ¥{pr.requested_amount}")
# 2. 各明细累计付款不能超过其应收金额
for pri in pri_rows:
    item = ...
    if (item.paid_amount or 0) + add > (item.received_amount or 0) + 0.005:
        raise HTTPException(400, f"明细「{item.item_name}」累计付款将超过应收金额")
```

### 🟡 中严重度（9项）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| M1 | **采购单号并发竞态** | `purchase_mgmt_router.py:661-678` | 两请求同时查到 `mx=0`，都生成 `TH20260721-001`。`po_no` 仅 `index=True` 无 `unique=True`，高并发下同号可静默共存 |
| M2 | **删除供应商竞态窗口** | `purchase_mgmt_router.py:174-188` | 两次 `count` 查询之间另一请求可能插入 PurchaseItem，应将检查+删除放在事务锁内 |
| M3 | **金额字段缺负数校验** | `purchase_mgmt_router.py:1459-1509` | `received_amount`/`invoice_amount`/`paid_amount` 未校验非负，汇总报表直接 sum，负数会导致数据错乱 |
| M4 | **custom_values 无类型校验** | `purchase_mgmt_router.py:822-838` | `_clean_custom` 不验证 number 是否真为数字、select 值是否在 options 范围内、date 是否合法 |
| M5 | **补价回写未排除已冲红单** | `purchase_mgmt_router.py:1487-1492` | 补价逻辑未排除 `reversed=True` 的原单，如果原入库已被冲红，补价仍会更新原单金额，导致冲红反向流水与原单不对等 |
| M6 | **批量收货无原子性** | `purchase_mgmt_router.py:1706-1714` | 循环逐条 `_finish_receive`，中间失败可能导致部分入库（依赖单一 commit） |
| M7 | **累计付款无上限** | `purchase_mgmt_router.py:2299-2304` | 多次请款付款后 `paid_amount` 可超过 `received_amount`，无累计校验 |
| M8 | **权限前端硬编码** | `PurchaseView.vue:29-31` | `isFangbusen`/`isWangqin`/`isLixinxin` 直接比 username，人员离职/换岗必须改代码发版 |
| M9 | **seeAll 未排除新采购员** | `PurchaseView.vue:32` | 新增采购员（非三人之一）也会获得全列权限，与预期分工矛盾 |

### 🟢 低严重度（6项）

| # | 问题 | 位置 | 说明 |
|---|------|------|------|
| L1 | **供应商 code 重复依赖 DB 异常** | `models.py:577` + `purchase_mgmt_router.py:116-144` | 代码层不预检重名，异常信息对用户不友好（500 而非「编码已存在」） |
| L2 | **导入供应商未设 created_by** | `purchase_mgmt_router.py:1298` | 导入的供应商对受限采购员不可见（`_buyer_restricted` 按 `created_by` 过滤） |
| L3 | **请购单删除无状态限制** | `purchase_mgmt_router.py:2798` | 已处理/已驳回的请购单仍可删除，破坏审计轨迹 |
| L4 | **驳回原因非必填** | `purchase_mgmt_router.py:2776` | 驳回时原因可为空，建议必须填写 |
| L5 | **打包下载 blob 错误处理** | `PurchaseView.vue:151-152` | catch 吞掉错误详情，无法区分 403/500/超时 |
| L6 | **付款完成后未通知采购申请人** | `purchase_mgmt_router.py:2166-2180` | 创建请款单时通知财务，但付款完成后未通知采购员 |

## 三、风险矩阵

| 风险 | 严重度 | 影响 | 建议 |
|------|--------|------|------|
| 超额付款无拦截 | 🔴 高 | 资金损失 | 付款接口加金额上限校验 |
| 采购单号重复 | 🟡 中 | 串单、对账混乱 | `po_no` 加唯一约束或用 DB 序列 |
| 补价修复已冲红流水 | 🟡 中 | 库存金额失真 | 补价时排除 `reversed=True` 的记录 |
| 前端权限可绕过 | 🟡 中 | 越权查看数据 | API 层按角色裁剪响应字段 |
| _BUYER_SHEET_MAP 硬编码 | 🟢 低 | 维护成本 | 迁移到数据库配置 |
| 成套收货不拆零件 | 🟢 低 | 业务限制 | 确认业务预期，如需独立出库则改设计 |

## 四、代码亮点（值得保留的设计）

1. **删除保护链**（`purchase_mgmt_router.py:1527-1537`）：删除采购明细前检查对账→流水→请款三道关卡
2. **自动对账**（`purchase_mgmt_router.py:628-640`）：现金全款+已收货+已付清自动置「已对账」，条件严格不误触
3. **内控职责分离**（`purchase_mgmt_router.py:2224-2227, 2274-2275`）：不能审批自己的请款单，审批人不能付款
4. **幂等设计**（`purchase_mgmt_router.py:1135-1138`）：防重复入库/冲红
5. **先收货后补价闭环**（`purchase_mgmt_router.py:1484-1492`）：补价后自动回写 WhTxn
6. **预计到货改期留痕**（`purchase_mgmt_router.py:1475-1507`）：改期自动回写清单+通知主管
7. **负库存保护**（`warehouse_router.py:787-791, 736-739`）：冲红/出库前校验现存够扣
