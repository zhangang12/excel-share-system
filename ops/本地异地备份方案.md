# 文件/数据 定时备份到本地服务器（异地冷备方案）

> 目标：把生产服务器（阿里云 8.141.123.141）的**上传文件（uploads）+ 数据库**每天自动拉一份到**本地服务器**留存，防云端误删/磁盘损坏/勒索/账号风险。

## 一、整体架构（两段式）

```
            ┌─────────────── 阿里云生产服务器 ───────────────┐
每天 06:25  │  ops/backup.sh                                  │
（已装cron） │   ├─ pg_dump        → /backup/pms-db-*.sql.gz   │
            │   └─ tar uploads卷  → /backup/pms-uploads-*.tar.gz (= 全部上传文件) │
            └───────────────────────┬─────────────────────────┘
                                     │ SSH 免密 + rsync/scp（本地主动拉，pull）
每天 07:30   ┌──────────────────────▼─────────────────────────┐
（本地cron） │  本地服务器  ops/pull-backup-to-local.sh        │
            │   → 拉到 /data/pms-backup/ ，保留 90 天，gzip 校验 │
            └─────────────────────────────────────────────────┘
```

- **为什么用 pull（本地拉）不用 push（云端推）**：本地服务器通常在内网/无公网 IP，pull 模型由本地主动发起出站 SSH，云端不用开任何端口、不用知道本地地址，最省事最安全。
- **文件数据在哪**：所有上传文件（合同/发票/图纸包/采购清单/截图…）都在 Docker 卷 `uploads_data`（容器内 `/app/uploads`，已于 a45e22a 修正路径）。`backup.sh` 第 2 步会把整个卷打成 `pms-uploads-*.tar.gz`，**这一个文件就是全部文件数据**。

## 二、前置：本地 → 云端 免密 SSH（一次性）

在**本地服务器**上执行：
```bash
ssh-keygen -t ed25519                       # 若还没有密钥，一路回车
ssh-copy-id root@8.141.123.141              # 公钥送到云端(输一次云端密码)
ssh root@8.141.123.141 "echo ok"            # 验证：能免密回显 ok 即成功
```
Windows 本地（PowerShell）：
```powershell
ssh-keygen -t ed25519
type $env:USERPROFILE\.ssh\id_ed25519.pub | ssh root@8.141.123.141 "cat >> ~/.ssh/authorized_keys"
ssh root@8.141.123.141 "echo ok"
```

## 三、确认云端每天在产备份（一次性）

登录云端，确认 `setup-cron.sh` 已装（生产部署时应已执行）：
```bash
ls -l /etc/cron.daily/pms-backup          # 存在即每天自动备份
# 没装就装：  sudo bash /root/excel-share-system-main/ops/setup-cron.sh
# 手动跑一次验证：  bash /root/excel-share-system-main/ops/backup.sh && ls -lh /backup
```

## 四、本地定时拉取

### A. 本地是 Linux / NAS / 群晖（用 pull-backup-to-local.sh）

手动验证一次：
```bash
REMOTE=root@8.141.123.141 LOCAL_DIR=/data/pms-backup \
  bash ops/pull-backup-to-local.sh
```
装每日 cron（本地 07:30，错开云端 06:25 备份完成之后）：
```bash
crontab -e
# 加一行（把路径换成实际的 ops 目录）：
30 7 * * * REMOTE=root@8.141.123.141 LOCAL_DIR=/data/pms-backup bash /opt/pms/ops/pull-backup-to-local.sh >> /var/log/pms-pull.log 2>&1
```
> 想"拉取前先在云端临时生成最新备份"，加 `RUN_REMOTE_BACKUP=1`（这样不依赖云端 cron 的时间点）。

### B. 本地是 Windows 服务器（用 pull-backup-to-local.ps1）

手动验证一次（PowerShell）：
```powershell
.\ops\pull-backup-to-local.ps1 -Remote root@8.141.123.141 -LocalDir D:\pms-backup
```
用「任务计划程序」装每日任务（管理员 PowerShell）：
```powershell
$action  = New-ScheduledTaskAction -Execute "powershell.exe" `
  -Argument '-NoProfile -ExecutionPolicy Bypass -File "D:\pms\ops\pull-backup-to-local.ps1" -Remote root@8.141.123.141 -LocalDir D:\pms-backup'
$trigger = New-ScheduledTaskTrigger -Daily -At 7:30am
Register-ScheduledTask -TaskName "PMS备份拉取" -Action $action -Trigger $trigger -RunLevel Highest -Description "每日从生产拉取数据库+上传文件备份到本地"
```

## 五、保留策略

| 位置 | 脚本 | 默认保留 | 调整 |
|---|---|---|---|
| 云端 /backup | backup.sh | 30 天 | `KEEP_DAYS=14 bash backup.sh` |
| 本地 | pull-backup-to-local.* | 90 天 | 改 `KEEP_DAYS` / `-KeepDays` |

本地建议比云端留更久（异地冷备的意义）。磁盘够的话本地可设 180/365。

## 六、恢复（演练 & 真出事时）

**恢复数据库**（在云端，或任意装了同版本 postgres 的机器）：
```bash
gunzip -c pms-db-YYYYMMDD_HHMMSS.sql.gz | \
  docker compose -f docker-compose.prod.yml --env-file .env.prod exec -T postgres psql -U pms_prod -d pms
# 仓库已有 ops/restore.sh 可直接用
```
**恢复上传文件**（把 tar 内容还原进 uploads 卷）：
```bash
UPLOADS_VOL=$(docker volume ls -q | grep uploads_data$)
docker run --rm -v "$UPLOADS_VOL":/dst -v "$PWD":/src alpine \
  sh -c "cd /dst && tar xzf /src/pms-uploads-YYYYMMDD_HHMMSS.tar.gz"
```
> 建议**每季度做一次恢复演练**：把某天的本地备份在一台测试机上还原，确认能开出文件。备份能恢复才算数。

## 七、校验与告警

- `pull-backup-to-local.sh` 每次拉取后会对最新 db/uploads 做 `gzip -t` 完整性校验，失败退出码 2。
- 建议本地 cron 行结尾接一个失败通知（邮件/钉钉/企业微信 webhook），例如：
  ```bash
  ... pull-backup-to-local.sh || curl -s "你的告警webhook" -d "PMS本地备份拉取失败 $(date)"
  ```

## 八、一句话总结

云端已每天把"数据库 + 全部上传文件"打包到 `/backup`；本地只要**配一次免密 SSH + 一条每日 cron 跑 `pull-backup-to-local`**，就能把这份打包每天拉回本地留 90 天，形成"云端 30 天 + 本地 90 天"双保险，且本地备份带完整性校验。
