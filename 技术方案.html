二、数据模型设计（新增表）

全部沿用 PortableJSON / 异步 Base；create_all 自动建表。
dept_orders          部门任务单（下单核心）
  id, project_id FK, dept(design/electric/produce),
  assignee_id FK users, created_by FK users,
  requirement Text, status(pending/doing/done),
  plan_start Date, plan_due Date, done_at Date,
  notify_user_id FK users,                 # 完成后通知谁
  is_overdue Bool, overdue_notified Bool,  # 逾期检测去重
  created_at, updated_at

attachments          通用附件（下单资料 + 完成产物 + 发货单）
  id, owner_type(dept_order/dept_order_output/shipment_invoice),
  owner_id, kind(material/word/nameplate_cad/purchase_list/circuit/invoice),
  stage(on_assign/on_complete),            # 采购清单=接单后；电路图/说明书/铭牌=完成时
  to_dept(purchase/logistics/null),        # 产物推送目标
  filename, stored_name, content_type, size,
  uploaded_by FK users, created_at

shipments            物流发货待办（每项目一条）
  id, project_id FK unique, status(pending/shipped),
  receiver_name, receiver_phone, receiver_addr,
  shipped_at, invoice_attachment_id FK attachments,
  created_at, updated_at
  # 汇总资料 = attachments(to_dept=logistics 且同 project)，按来源部门分三栏

notifications        站内消息 + 企微发送记录
  id, to_user_id FK, kind(wx/overdue/approval),
  title, content, link, is_read,
  channel(inapp/wechat), wechat_sent Bool, created_at

export_requests      导出权限申请/审批（一次性）
  id, requested_by FK, scope, status(pending/approved/rejected),
  approver_id FK, decided_at, created_at

字段/小改动

  • roles：seed 增加 electrician / assembly_1 / assembly_2 / assembly_3 / logistics / purchasing / design_lead / electric_lead / produce_lead。
  • roles 加 is_push_target Bool + 可选 dept（逾期按部门推主管）。
  • users 加 wechat_userid String（企微 touser；为空则仅站内）。
  • 四表上传状态不新增字段，派生：查 4 个固定 Datasheet 是否各有 ≥1 条 Record。

五、关键技术点

  1. 角色扩展：改 seed.ROLES；roles 加 is_push_target；迁移 backfill_new_roles（幂等）。
  2. 附件存储：落盘 /app/uploads/{yyyymm}/{uuid}{ext}（卷已挂），DB 存元数据，按归属鉴权下载；量大可平滑换 OSS。
  3. 四表校验：GET sheet-status 查 4 表 Record 数；设计部 complete 缺表返回 422。
  4. 产物上传与推送（分阶段）：① 电工部接单后(stage=on_assign)即可传采购清单(Excel)→采购部收件箱，不必等完成；② 设计部完成传 说明书(Word)+铭牌(CAD)→物流；③ 电工部完成时强制校验电路图(PDF)已上传，否则 complete 返回 422；产物 attachments.to_dept 决定去向，complete 事务内归集并发通知。各部门「完成必传产物」由 DEPTS 配置 required 声明。
  5. 企业微信：wechat.py：get_token(Redis 缓存) + send_text/send_card；.env 配 WECHAT_*；凭证未到位整体降级仅站内；失败不阻断主流程。
  6. 逾期检测：lifespan 起 asyncio 定时（每日 09:00）+ Redis SETNX 锁；扫 doing & plan_due<today & !overdue_notified → 推对应主管 → 标记。
  7. 站内通知：ws_router 加 user:{id} room + broadcast_to_user；未读走 GET /notifications。
  8. 导出审批（一次性）：点导出→建 export_request(pending)→审批 approved 仅放行本次。
  9. 物流三栏 + 收货：shipments 持收货信息；汇总资料 = attachments(to_dept=logistics) 按 from 部门分组。

六、分期实施路线

阶段内容预估
P1 角色+下单+执行角色扩展；dept_orders/attachments；下单→待办→开始→完成(可逆)→效率；制图回传；四表校验~5–7 人日
P2 下游+物流产物推送采购/物流；采购收件箱；物流三栏+收货信息+发货~3–4 人日
P3 站内通知user:{id} 推送 + 消息中心 + 铃铛~2 人日
P4 企微+逾期企微消息 API；逾期定时推主管~3 人日
P5 报表+导出审批月度报表聚合；导出申请/审批；推送人开关~3–4 人日
合计 ~16–20 人日（不含联调与企微对接）。每阶段可独立上线（main 直推 + docker 重建 + run_all 自动迁移）。

七、风险与取舍

  • 角色非运行时可配：新增角色需改代码+重启（与现状一致，简单稳）。
  • 企微依赖外部凭证：P4 与凭证申请并行，先上站内通知版。
  • 定时任务多实例：单 backend 容器无虑；扩容靠 Redis 锁。
  • 附件落盘 vs OSS：先落盘（卷已挂），按量再迁。
  • 具体业务口径以《需求分析》第五节确认结果为准（影响逾期/效率/四表校验等实现细节）。

八、企业微信消息推送方案（详）

8.1 集成方式

采用企业微信「自建应用」+ 应用消息（message/send）：后端用应用 corpid + secret 换 access_token，向应用可见范围内成员推送。无需第三方/代开发。

8.2 前置准备（企业管理员一次性）

  1. 企业微信管理后台 → 应用管理 → 自建 → 创建应用，记录 AgentId / Secret；「我的企业」页拿 CorpID。
  2. 应用「可见范围」勾选需接收通知的部门/成员。
  3. 「企业可信IP」加入服务器公网出口 IP（调用发送接口要求）；如用消息内 H5 跳转，另配「可信域名」。

4.（可选）开启通讯录同步，便于系统用户↔企微成员自动映射。

8.3 用户映射（User ↔ 企微 UserID）

  • users 增 wechat_userid；为空者仅站内通知（自动降级）。
  • 映射方式（任选/组合）：手机号匹配（调通讯录接口回填，推荐）/ 管理页手动绑定 / OAuth 自助（企微内打开系统授权拿 userid）。

8.4 access_token 获取与缓存

  • GET /cgi-bin/gettoken?corpid=&corpsecret=，有效期 7200s。
  • Redis 缓存 wx:token:{agentid}，TTL 7000s（提前过期）；取 token 加 SETNX 锁防并发重复。
  • 收到 40014/42001（token 失效）→ 清缓存重取一次。

8.5 发送消息

  • POST /cgi-bin/message/send?access_token=，参数 touser(企微 userid，多人用 |)、agentid、msgtype。
  • 按场景选类型：完成/审批 → textcard（标题+描述+跳转 url）；逾期 → markdown（富文本）；兜底 → text。
  • 跳转 url 指向系统页（如 {BASE_URL}/#/project/{id}）；企微内免登可配 OAuth2 网页授权。

8.6 后端实现（app/wechat.py）

async def get_access_token() -> str          # Redis 缓存 + 锁 + 自动刷新
async def send_textcard(touser, title, desc, url)
async def send_markdown(touser, content)
async def send_text(touser, content)

与站内通知合并的统一入口：

async def notify(user, *, title, content, link=None, kind):
    1) 写 notifications（站内，必做）
    2) if settings.wechat_enabled and user.wechat_userid:
         try:  send_textcard/markdown; wechat_sent=True
         except: wechat_sent=False        # 不抛、不阻断主流程
    3) ws.broadcast_to_user(user.id, ...)  # 铃铛

触发点：dept_orders.complete（通知所选人）、逾期定时（通知部门主管）、export_requests 审批结果。

8.7 配置（.env）

WECHAT_ENABLED=true
WECHAT_CORPID=ww************
WECHAT_AGENTID=1000002
WECHAT_SECRET=********************
WECHAT_BASE_URL=https://你的系统域名     # 生成跳转链接

config.py 加对应字段；WECHAT_ENABLED=false（凭证未到位/开发）→ 全走站内通知。

8.8 容错 / 限流 / 重试

  • 发送 try/except，失败不影响主业务，失败记 notifications.wechat_sent=false。
  • 企微有频率配额：批量逾期推送串行+间隔或合并多 touser 一条。
  • 可选补发：定时扫 wechat_sent=false 近 N 小时记录重试。

8.9 安全

  • Secret 仅放环境变量，不入库、不下发前端、不写日志。
  • 「企业可信IP」白名单 = 服务器公网 IP。

8.10 联调步骤

  1. 管理员建应用、配可见范围与可信IP。
  2. 配 .env → 调试脚本先给自己 userid 发一条 text 验证打通。
  3. 回填测试用户 wechat_userid → 跑「完成→通知」真实链路。
  4. 灰度：先设计部启用，再全量。

8.11 时序（完成通知）

设计师点完成 → complete 事务（写完成/产物推送）
   → notify(通知人)：写 notifications → get_token(Redis) → message/send(textcard)
   → ws 推铃铛
通知人企微收到卡片 → 点开（OAuth 免登）→ 系统对应页
对接现有系统：复用 notifications(channel/wechat_sent)、users.wechat_userid、ws_router 的 user:{id} 推送；触发点在 complete/逾期任务/审批。WECHAT_ENABLED=false 时 P3 站内先行、P4 接通企微。
