4. **性能**：新列表端点一律分页；聚合查询加索引；不复制现有无分页的坑。
5. **UI 风格**：新页面统一 element-plus + 现有 Excel 皮肤；不重构老页面样式。

## 开发顺序（含理由）

1. M01 角色与权限扩展 —— 一切的地基：新角色/菜单矩阵/路由守卫/详单闸门/附件基建/采购合并迁移，后续所有模块的角色判定与文件能力都依赖它；先行可让各角色账号提前建好供联调
2. M15(基建部分) 消息中心 —— messages 表+push_message 角色池服务+铃铛最小版先落（虽整体 P2，但 M02/M04/M08/M10 每个流程节点都要推送，不先行则处处打桩）；企微对接与逾期定时任务可滞后到 P2 阶段补
3. M11(制图列部分) 项目目录增强 —— 制图三列逻辑删除是最高敏感独立项，纯展示层过滤+导出同步，无依赖、风险集中、先做先验收；下单入口/收货信息子项推迟到 M04/M08 就绪后联调
4. M04 部门工作台+派单接单完成流 —— 全系统核心状态机与最复杂事务（orders 模型/DEPTS 配置/完成-撤回-作废/D1 导入标记/效率口径），是 M02 下单、M05/M06 下游推送、M08 闸门、M14 报表、M17 换人的共同上游，必须最早进入联调
5. M02 销售部台账+下单 —— 业务流程入口：建项目+建任务+建发货待办一体事务，依赖 M04 的 orders 模型；台账与 ledger 回填迁移在此落地，打通'销售下单→部门接单'第一段链路
6. M12 项目详单增强 —— SHEET_TEMPLATES 加第 5 表+存量补表迁移+协作 tab+工作流聚合端点；紧跟 M04 因电工上传清单要落第 5 表，且 workflow 端点被销售全览复用；存量补表迁移需尽早上线观察
7. M05 钣金组 + M06 采购部 —— 两个轻量下游承接页可并行开发：均只消费 M04 推送的附件/收件箱数据，工作量小、验证'接单即传'链路完整性
8. M07 仓库组 —— 相对独立的大模块（三表+六 tab+清洗工具），仅依赖 M01 角色，可与 M05/M06 并行排人；其发货清单上传是 M08 看板的输入，须在 M08 联调前完成 W1-W3 分期
9. M08 物流发货 —— 闸门是全链路汇聚点，依赖 M04 任务状态+M07 发货清单+M02 收货信息/回传目标；shipments 存量补建迁移与零任务单兜底策略在此验证，打通'部门完成→发货→回传销售'第二段链路
10. M03 开票流 + M09 财务部 —— 资金线收尾：开票状态机依赖 M02 ledger，财务页面承接 M03 上传发票；两者连排一个迭代，至此 P1 主干（销售→部门→物流→财务）全通，可做端到端验收
11. M10 售后部 —— P2 之首：独立闭环且财务售后 tab 已留位，依赖少可随时插入
12. M13 问题反馈流 —— 依赖 M04 派工关系与 M12 协作 tab 存档容器，在工作台稳定后叠加
13. M17 换人 —— M04 之上的单端点叠加，量小；放在报表前做以便效率归属口径一并确认
14. M16 导出审批 —— 涉及给老导出端点加闸（唯一动老接口项），放后期且开关默认关闭灰度，待业务确认后开启；PDF 能力在此引入供报表复用
15. M14 报表 —— 最后做：纯聚合消费 M02/M04/M07 数据，数据越全数字越可信；三报表口径函数复用 M04 已沉淀的效率/按时服务，开发成本最低风险最小
16. M15(剩余部分) 企微对接+逾期每日提醒 —— 凭证到位后接入，站内兜底已运行多时，切换无感；至此全清单收口

## 横切关注点

- 角色体系扩展：维持单角色制+role.code 承担部门语义（不建 Department 表，最低成本路径）；seed.ROLES 幂等同步；新增 require_roles 依赖工厂统一接口守卫；保持 admin/manager 全权与 admin 隐身两条历史约定；警惕'新建项目全员 edit 成员/新用户入全项目'两个自动化机制与新角色行级隔离的冲突——可见性收紧必须在菜单+接口+数据出口三层同时落地
- 菜单权限：后端 GET /api/me/menus 为唯一权威，前端 MainLayout 手写导航按其渲染（不引入 el-menu，保持现有写法）；2026-06-12 最新收紧口径为准：销售/电工/装配组只留本部门菜单+项目目录且编号不可点详单、售后部无项目目录、elead 无详单而 dlead 有（不对称是有意设计）、月度报表与管理组菜单仅管理层（部门主管也无权）
- 附件基建（全局新增）：统一 attachments 表+上传/下载端点+data/files 存储，合同/开票申请/发票/图纸包/说明书/铭牌/电路图/采购清单/发货单/发货清单/售后物料清单全部复用；下游撤回/移除一律按附件 ID 关联（修正原型按文件名匹配的缺陷）；下载沿用前端 fetch+Authorization+blob 模式
- 消息通知：messages 表+push_message(to_user|to_role) 服务先行（虽属 P2 模块但为 M02/M04/M08/M10 推送依赖，基建部分在 M01 后立即落地）；按角色池路由替代原型硬编码个人（lo1/sm1/fin1-3）；企微 F1-F3 口径：手动绑 userid、凭证未就绪降级站内不阻塞、逾期每日提醒一次且幂等键防多实例重复
- UI 风格统一：新页面一律 el-table+现有 Excel 皮肤 :deep 样式块（建议抽成 excel-table.css 共享避免第三次复制）、page-header 全局类、对话框 width 500-560/label-top/size large、ElMessage/ElMessageBox 既有用法、localStorage 全 pms_ 前缀；不使用已注册未用的 vxe-table（仓库月度大矩阵为唯一例外候选）；前端验证用 npm run build（本地连不通后端）
- 状态与口径常量统一：dept_orders 状态英文枚举+中文映射单一来源；效率/按时/逾期口径函数（C1-C5）服务层单一实现供完成流/报表/逾期推送共用；日期一律 normalize_date_str→YYYY-MM-DD（新表'日期'结尾字段自动命中，非常规命名加 _DATE_FIELD_EXTRA）；项目编号 P-21 后端唯一校验+年份参数化
- 实施红线（贯穿所有模块）：只增不改——新功能全部新菜单/新表/新接口/新组件；动老页面仅三处且均可逆：①OverviewView/DatasheetGrid 制图三列过滤（hidden 标记可改回，无运行时开关是业务口径）②excel 导入全量替换白名单保护第 5 表③导出审批闸 settings 开关默认关；ProjectsView/ProjectDetailView 原 tab/DatasheetGrid 内部/OverviewView 既有列零重构
- 存量迁移纪律：所有迁移写成幂等函数（查现状→只补缺失/只改命中）追加 data_migration.run_all 末尾，顺序敏感且单函数失败仅 warn；本期迁移清单：采购合并(P-22)/sales_ledger 回填/datasheets.imported_at 回填/第 5 表补建/shipments 补建；仓库主数据+期初导入因需人工清洗确认改走管理端工具不挂 run_all；SHEET_TEMPLATES 加'电工采购单'前必须确认线上无同名存量表（cleanup 对齐会清数据）
